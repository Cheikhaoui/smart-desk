# SmartDesk Backend — Project Guide for Claude

## Project overview
SmartDesk is an AI-powered helpdesk/ticketing backend. Tickets are created by users,
assigned to agents, and enriched with AI features (auto-categorization, reply
suggestions, thread summaries). This project is my job-market portfolio piece —
aim for senior-level code at all times.

## Stack
- Java 21 LTS (use modern features: records, pattern matching, virtual threads where relevant)
- Spring Boot 3.3.5
- PostgreSQL 16 (UUIDs, not Long ids)
- Redis 7 (sessions, caching, rate limiting — later)
- Flyway 10 for migrations
- JWT auth via jjwt 0.12.6
- Spring Security (stateless)
- springdoc-openapi 2.6.0 (Swagger UI + OpenAPI spec)
- Lombok
- Testcontainers for integration tests

## Architecture — vertical slice, NOT layered
Organize by feature, not by technical layer. Each feature package is self-contained.

```
dev.smartdesk/
├── SmartDeskApplication.java
├── auth/         JWT, login, register, JwtService, JwtFilter
├── user/         User entity (UserDetails), repository, DTOs
├── ticket/       Ticket + Comment entities, repositories, DTOs, service, controller
├── notification/ WebSocket / STOMP for real-time events
├── ai/           AI layer — AiProvider interface + implementations
├── config/       SecurityConfig, OpenApiConfig, WebSocketConfig
└── common/       BaseEntity, ErrorResponse, GlobalExceptionHandler
```

NEVER create `controllers/`, `services/`, `repositories/` folders. Each feature
owns all its layers.

## Code conventions (enforce these strictly)
- **Records for DTOs**, classes for entities. Never mix.
- **Static `from(entity)` factory methods** on DTOs for mapping. No MapStruct for this size.
- **UUIDs** for all primary keys, generated by Postgres (`uuid_generate_v4()` or
  `gen_random_uuid()`).
- **BaseEntity** (`@MappedSuperclass`) provides `id`, `createdAt`, `updatedAt`.
  Entities extend it unless they don't need audit fields (Comment is the exception).
- **Flyway owns the schema.** `spring.jpa.hibernate.ddl-auto: validate`, never
  `update` or `create`.
- **Dynamic search queries** use the `:param IS NULL OR field = :param` JPQL pattern.
- `@Transactional(readOnly = true)` at class level, write methods annotated individually.
- **Permission checks in the service layer**, not controllers. Controllers are thin.
- **Stateless sessions**, CSRF disabled, JWT in `Authorization: Bearer` header.
- **Immutable events** (records) for ApplicationEventPublisher.
- **Loose coupling**: services publish events, listeners consume them. No direct
  dependency from TicketService to WebSocket or AI layers.
- Use `@AuthenticationPrincipal User currentUser` to get the caller, never read
  `SecurityContextHolder` manually in controllers.
- Return `ResponseEntity` only when you need to set status codes (e.g. 201 Created);
  otherwise return the DTO directly.
- BCrypt password encoder with cost factor **12** (2026 baseline).

## Naming
- Packages: lowercase single word (`auth`, `ticket`, not `authManagement`)
- DTOs: inner records inside a class named `<Feature>Dto` — e.g.
  `TicketDto.CreateRequest`, `TicketDto.Response`, `TicketDto.Summary`
- REST paths: `/v1/<resource>`, kebab-case if multi-word
- Database: `snake_case` tables and columns; Java stays camelCase, JPA maps via
  `@Column(name = "...")`

## How to run

### Local dev (infrastructure only, app on host)
```bash
docker compose up -d postgres redis pgadmin
./mvnw spring-boot:run
# Swagger UI → http://localhost:8080/api/swagger-ui.html
```

### Full stack in Docker (app + infra)
```bash
# Optional: set your API key so AI features activate
export ANTHROPIC_API_KEY=sk-ant-...

docker compose up -d --build
# Swagger UI → http://localhost:8080/api/swagger-ui.html
```

### Useful env vars (override via shell export or .env file)
| Variable | Default | Required in prod |
|---|---|---|
| `JWT_SECRET` | weak dev default | **YES** — use 256-bit random |
| `ANTHROPIC_API_KEY` | empty (AI disabled) | No |
| `CORS_ALLOWED_ORIGINS` | http://localhost:4200 | YES — set to your frontend URL |

## OpenAPI / API contract — frontend integration rules

The Angular frontend (`smartdesk-frontend`) consumes this backend via
`ng-openapi-gen`, which fetches `http://localhost:8080/api/v3/api-docs`
and generates a typed TypeScript client. The frontend treats the
generated client as the source of truth — there is no manual API
wrapper. **If the spec is wrong, the frontend is wrong.**

### Hard rules (any violation breaks the frontend)

1. **The spec endpoint must be publicly accessible.**
   In `SecurityFilterChain`, allowlist:
   `/v3/api-docs/**`, `/swagger-ui/**`, `/swagger-ui.html`

2. **Never rename or remove a field from a response DTO** without a frontend change.
   Adding optional fields is safe; removing or renaming fields is a breaking change.

3. **Enum values in responses must be stable.**
   `Ticket.Status` and `Ticket.Priority` values are used as TypeScript string literals.
